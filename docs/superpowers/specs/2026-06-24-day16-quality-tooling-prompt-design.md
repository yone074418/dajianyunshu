# Day16 工程质量工具执行提示词设计

> 选择方案：Day16 独立强约束提示词
>
> 本设计只定义执行提示词，不执行 Day16 任务。

## 1. 目标与起点

执行代理必须基于最新远程 `main` 创建独立分支 `ai-codex/week3-day16-quality-tooling`，仅完成《126天计划》第16天任务：为 `heavy-transport-sim/` 配置 ESLint、Prettier、Vitest 和 Playwright，并证明各检查能够发现错误、修复后全部通过。

## 2. 范围

允许修改工程质量配置、`package.json`、锁文件、必要的最小测试文件和质量验证记录。不得实现第17天路由与全局布局，不得加入业务功能、Supabase、状态管理、3D、CI或部署配置，不得修改既有需求基线文档。

## 3. 工具职责

- ESLint：检查 TypeScript 与 React 代码质量，覆盖 React Hooks 和热更新导出规则。
- Prettier：统一源码、配置和 Markdown 的格式，并提供只检查不改写的命令。
- Vitest：配置浏览器式组件测试环境和最小冒烟测试，证明测试失败能阻断命令。
- Playwright：配置 Chromium 项目和最小页面冒烟测试，通过可控的本地 Vite 服务运行。

脚本名称应清晰稳定，至少提供 `lint`、`format`、`format:check`、`test`、`test:run`、`test:e2e` 和 `build`。如实际依赖版本要求不同，可做兼容调整，但必须记录原因，不能弱化检查。

## 4. 红—绿验证

执行代理须分别证明质量门禁有效：临时制造可归因的 ESLint、Prettier、Vitest 和 Playwright 失败，运行对应命令并保存非零退出码及关键错误，再恢复正确内容并重新运行。故意失败内容不得提交，验证后工作树中不得残留破坏性改动。

最终全绿至少包括：格式检查、Lint、Vitest 非交互运行、Playwright 端到端测试和生产构建。不得以旧日志、局部检查或“应该通过”代替当前分支上的实际命令结果。

## 5. Git 与安全边界

执行前获取最新远程状态并检查工作区。发现无关修改时不得删除、覆盖、暂存或 stash；应采用安全隔离方式，无法隔离则停止报告。不得强推、改写历史、合并 `main` 或创建 PR。

只暂存 Day16 文件，使用提交信息 `完成第3周第2天工程质量工具配置`，推送 Day16 分支。提交前必须确认故意失败内容、运行日志、浏览器下载缓存、测试报告、截图和构建产物未被误提交。

## 6. 环境与失败处理

先记录 Node、npm 和操作系统环境。安装依赖或 Playwright Chromium 失败时保留原始错误；允许在 Day16 范围内修复版本或缓存问题，但不得跳过端到端测试、改用虚假测试或声称全绿。开发服务器必须由 Playwright 配置可靠启动和停止，不得留下后台进程。

## 7. 完成判定

只有同时满足以下条件，才能判定 Day16 完成：

- 四类工具均已配置并存在可重复执行的脚本；
- 四类故意失败均产生预期非零结果，且错误随后被修复；
- 格式检查、Lint、Vitest、Playwright 和生产构建在最终状态全部退出为零；
- Git 差异仅包含 Day16 范围文件，没有生成物、凭据或无关修改；
- 独立提交和分支推送成功，未执行 Day17、未创建 PR、未合并 `main`。

## 8. 最终回报

最终只回报分支、提交哈希、推送结果、变更文件、工具与脚本、四类红—绿证据、最终全绿命令结果、未执行事项和剩余风险。任何失败必须按实际状态报告，不得为了提交而缩减验收范围。
