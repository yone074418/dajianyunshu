# Day14/Day15组合执行提示词设计

> 选择方案：一份提示词、两个串联分支、两个独立提交
>
> 目标：生成可直接交给执行代理的强约束提示词，顺序完成Day14与Day15；本设计不执行两天任务本身

## 1. 总体执行模型

执行提示词分为两个不可倒置的阶段。Day14先完成G1正式评审；只有Day14明确通过、发布`BL-001/v1.0.0`冻结快照并推送成功后，才允许创建Day15分支。Day14未通过、证据不完整、提交或推送失败时，立即停止，不得创建Day15分支或工程文件。

## 2. Git链与提交

1. Day14分支：`ai-codex/week2-day14-g1-freeze`，基于远程Day13分支`origin/ai-codex/week2-day13-overall-acceptance-checklist`的最新提交。
2. Day15分支：`ai-codex/week3-day15-vite-react-scaffold`，仅在Day14通过后，基于已推送的远程Day14分支创建。
3. Day14通过提交信息：`完成第2周第7天G1需求冻结`。
4. Day14未通过提交信息：`记录第2周第7天G1冻结评审未通过`；推送后终止。
5. Day15提交信息：`完成第3周第1天Vite React TypeScript工程初始化`。
6. 两个分支分别推送；不创建PR、不合并main、不改写历史、不强推。

## 3. Day14产物与边界

Day14唯一业务产物为`docs/G1需求冻结评审记录.md`。不得回写Day1—13历史文档，不得编写代码、安装依赖或执行Day15。

评审记录必须：

- 完整读取Day1—13候选材料、PCR/PDEC/CBL证据、Git历史和Day13总体验收清单；
- 核对当前54个页面/界面条目、六阶段、15状态、54转换、15非法转换、25个SM-AC、44条专业规则、26项评价、14个数据/逻辑对象及AR-001—019；
- 区分Day7历史结论、PCR追加式关闭、Day14正式G1结论；
- 逐项审查G1五个域：页面、六阶段流程、专业判定契约、数据设计、资产需求基线；
- 证明专业具体案例值仍按Day50—91形成，资产来源/许可/预算仍按Day29形成；
- 将Day13的NFR、PERF-BUDGET和资产文件级追踪问题保留给后续门，不冒充G1阻断或已冻结性能阈值；
- 给出唯一结论“通过”或“未通过”，列明证据、责任人、关闭条件和阻断项。

若通过，文件必须建立`BL-001/v1.0.0`正式冻结记录，包含冻结提交、文件清单、内容哈希、来源CBL、批准结论、冻结范围、未冻结范围、变更入口和签核。若未通过，不得生成BL编号或`v1.0.0`已发布结论。

## 4. Day14通过门

只有同时满足下列条件，才允许进入Day15：

- Day13远程提交和唯一产物可验证；
- G1五个域不存在未关闭P0或无责任/关闭条件的阻断项；
- 固定数量、编号集合和关键语义在最新候选材料中一致；
- Day7历史不回写，PCR/CBL不冒充正式BL；
- G1冻结的是专业判定契约和资产需求，而不是后续案例值、资产文件或性能预算；
- `docs/G1需求冻结评审记录.md`明确通过并形成`BL-001/v1.0.0`快照；
- Day14提交与推送成功，远程分支提交与本地HEAD一致。

## 5. Day15工程位置与边界

Day15在仓库根目录下创建`heavy-transport-sim/`，复用现有Git仓库，禁止执行`git init`或创建嵌套`.git`。不得覆盖根目录既有论文、文档、脚本和资料。

使用Vite的React TypeScript模板初始化工程，安装模板所需依赖，并验证`npm run dev`可启动、HTTP入口可访问、`npm run build`成功。记录实际Node、npm、Vite、React和TypeScript版本及验证证据。

Day15只建立工程底座：

- 允许模板生成的`package.json`、锁文件、`src/`、`public/`、TypeScript和Vite配置；
- 允许保留模板自带的基础ESLint文件，但不得把它描述为Day16质量配置完成；
- 不得主动配置Prettier、Vitest、Playwright、CI、路由、权限、Supabase、状态机、3D场景或业务功能；
- 不得执行Day16及以后任务，不得提前宣称G2或任何业务验收完成；
- 不得加入密钥、环境凭据、真实案例值或额外依赖。

## 6. Day15验证和失败处理

验证顺序固定为：确认目录边界→初始化模板→安装依赖→检查脚本和文件集合→执行生产构建→启动开发服务器→轮询本地HTTP入口→停止服务器→检查无残留进程→检查Git差异。

任何一步失败必须保留原始错误并停止提交；允许在Day15范围内修正脚手架或环境兼容问题后重试，但不得通过增加业务依赖、跳过命令或伪造输出绕过。开发服务器必须后台启动并在验证后可靠停止。

## 7. 提示词完成标准

- 一份提示词可从Day13远程成果开始连续执行两天；
- Day14和Day15使用两个串联分支、两个提交和两个推送结果；
- Day14未通过时能安全停止且不会产生Day15内容；
- Day14正式冻结与PCR候选证据、后续专业配置、资产制作和性能预算边界清楚；
- Day15复用现有Git仓库，在`heavy-transport-sim/`创建工程且不污染根目录；
- `npm run dev`和`npm run build`均有实际证据；
- 不提前执行Day16、不创建PR、不合并main。
