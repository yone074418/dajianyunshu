## 1. 目标与边界

本文只解决GAP-002“预G1候选基线CR路径缺失”，并为后续CON-001、CON-006等整改建立合法入口。本文不宣布G1通过，不关闭CON-002、CON-004、GAP-001、GAP-003或GAP-004，不补造专业参数、评价口径或资产事实。

Day 7历史结论保持“复盘通过、G1预审不通过”。问题是否在Day 7之后关闭，由本文建立的整改台账追加记录，不回写历史审查结论。

## 2. 现状证据

1. 《范围排除清单与变更流程》§12要求所有CR关联“当前有效BL编号和业务版本”。
2. 同文§17规则1规定未关联有效BL的CR不得审批。
3. 同文§18规定只有G1通过后才建立`BL-001 / v1.0.0`。
4. Day 7 GAP-002据此认定：预G1整改既必须走CR，又不存在可关联的正式BL，形成闭环缺口。
5. 当前Day 12提交为`1c363a4`；Day4、Day8、Day9/10和Day11是从共同祖先独立演进的文档分支，尚未形成单一候选链。

## 3. 推荐治理方案

采用“候选基线CBL + 预G1变更PCR + 决策PDEC”三件套：

- `CBL-001`：只表示可审计的预G1候选输入集合，不是正式冻结基线，不具有G1通过含义。
- `PCR-001`起：用于G1前纠错、澄清、文档统一和阻断项整改；字段、角色、13状态、17维影响分析、专业/权限门禁和验收要求全部复用Day 6 CR规则。
- `PDEC-001`起：项目负责人对PCR作批准、拒绝、暂缓或要求补充资料的唯一决策记录。
- PCR批准后先产生新的CBL修订版，才能执行对应文档整合或修订；“批准”仍不等于可以跳过基线更新。
- 第14天正式G1通过时，使用最终CBL快照建立`BL-001 / v1.0.0`；所有PCR/PDEC作为来源历史保留。

该方案不修改正式CR/DEC/BL的既有语义，只补足它们在G1前的入口。

## 4. 编号、状态与版本规则

| 对象 | 编号 | 含义 | 是否正式G1基线 |
|---|---|---|---|
| 候选基线 | CBL-001、CBL-002…… | 某一时点的候选文档与提交清单 | 否 |
| 预G1变更 | PCR-001、PCR-002…… | G1前的受控整改申请 | 否 |
| 预G1决策 | PDEC-001、PDEC-002…… | PCR唯一决策事件 | 否 |
| 正式基线 | BL-001 / v1.0.0起 | 第14天G1通过后发布 | 是 |

PCR完整复用Day 6的13个状态：草稿、待完整性检查、待影响分析、待专业确认、待决策、已批准、已拒绝、已暂缓、已纳入基线、实施中、待验收、已关闭、已撤销。这里“已纳入基线”在G1前特指纳入CBL，不得写成纳入正式BL。

候选版本使用`c0.1.0`起的主.次.修订号。文档合链与不改业务语义的纠错至少升修订号；业务语义变化升次版本。正式G1不得直接把未决PCR带入`BL-001`。
