# 大件运输虚拟仿真实验教学系统复刻实施计划

> **面向执行者：** 实施本计划时，应逐日勾选任务并保存验收证据；进入编码阶段后，建议使用 `subagent-driven-development` 或 `executing-plans` 按阶段执行。

**目标：** 一个人在 126 个自然日内，复刻论文《大件运输虚拟仿真实验教学系统设计与实现》所描述的教学版系统，使学生可以通过浏览器完成一次完整的大件运输虚拟仿真实验，系统能够记录、评分，教师能够查看和评价。

**架构：** 采用浏览器端 3D 教学系统。React 负责学生端、教师端和实验流程界面；React Three Fiber/Three.js 负责三维场景；Rapier 仅承担教学所需的碰撞和刚体效果；Supabase 提供登录、PostgreSQL 数据库和模型资源存储；Netlify 承担前端部署。专业判断由独立规则引擎完成，不把工程规则散落在界面代码中。

**技术栈：** Vite、React、TypeScript、React Router、React Three Fiber、Drei、Rapier、Zustand、TanStack Query、Zod、Supabase、Vitest、Playwright、Netlify。

---

## 一、执行假设

- 每天投入 3-4 小时，每周 6 天开发，第 7 天只复盘、修补或休息。
- 首版只实现论文中的一套运输案例，不扩展为多案例平台。
- 三维模型以教学表达准确、浏览器运行流畅为标准，不追求影视级效果。
- 桥梁、弯道、坡度、轴载和绑扎判断均为教学规则，不宣称可替代真实工程校核。
- 每个工作日必须留下至少一种证据：提交记录、测试结果、截图、演示录像或验收表。
- 任一阶段验收未通过，不得因为赶进度跳到下一阶段。

## 二、论文功能映射

| 论文模块 | 复刻功能 | 完成判定 |
|---|---|---|
| 学生登录 | 学生/教师登录、角色权限 | 学生不能访问教师页面或他人成绩 |
| 知识学习 | 大件运输知识、车辆、路线、绑扎资料 | 学习进度可保存，实验中可查阅 |
| 操作提示 | 当前任务、步骤提示、错误反馈 | 每个关键步骤均有提示且可记录使用次数 |
| 操作日志 | 关键选择、正误、耗时、提示次数 | 刷新后日志不丢失，教师可查看 |
| 运输任务及货物介绍 | 任务背景、起终点、货物参数、360°模型 | 参数完整，模型可旋转缩放 |
| 简单配车 | 组合方式、牵引车、挂车、轴线数 | 规则引擎能解释正确和错误原因 |
| 路线勘测 | 高度、圆弧弯道、直交弯道、坡度、桥梁 | 三条路线均可测量并形成通行结论 |
| 车组确定 | 增减牵引车、调整悬架、拼接挂车、液压编点 | 路线结果会约束最终车组方案 |
| 货物装车及绑扎 | 重心放置、工具选择、顺序、绑扎点和角度 | 错误操作可回退并扣分 |
| 货物运输 | 路线复验、成功/失败动画 | 只有满足规则的方案可完成实验 |
| 学习评价 | 系统评分、AHP权重、教师评价 | 可生成分项成绩、总分和评语 |

## 三、严格范围

**本期必须做：** 一个案例、三条路线、六阶段实验、学生端、教师端、日志、自动评分、教师评价、成绩导出、正式网址。

**本期禁止做：** VR、多人同步、移动端适配、开放世界、真实有限元分析、复杂车辆动力学、多案例编辑器、即时聊天、排行榜、AI 辅导。

## 四、预定项目结构

```text
heavy-transport-sim/
  src/app/                 # 路由、权限、全局布局
  src/features/auth/       # 登录与角色权限
  src/features/learning/   # 知识学习
  src/features/simulation/ # 六阶段实验流程
  src/features/scoring/    # 日志、规则评分、教师评价
  src/features/teacher/    # 教师管理端
  src/scene/               # 3D场景、相机、模型、交互
  src/rules/               # 高度、弯道、坡度、桥梁、轴载、绑扎规则
  src/stores/              # Zustand状态机和会话状态
  src/services/            # Supabase与资源接口
  src/types/               # 领域类型
  public/models/           # 开发期低模GLB资源
  supabase/migrations/     # 数据库迁移与RLS策略
  tests/unit/              # 规则和状态机单元测试
  tests/e2e/               # Playwright主流程测试
  docs/                    # 规则手册、资产清单、测试与部署文档
```

## 七、每日结束检查

每天结束前必须完成以下检查：

- [ ] 今天的产物可以被别人打开或复现。
- [ ] 验收标准有证据，不以“应该可以”代替实际验证。
- [ ] 新规则已写入规则手册并有测试。
- [ ] 新数据结构有迁移和权限策略。
- [ ] 新页面至少覆盖加载、空数据、成功、失败状态。
- [ ] 新3D交互不会造成明显卡顿、穿模或无法退出。
- [ ] 变更已提交，提交信息能说明完成了什么。
- [ ] 明天第一项任务已经明确，不需要重新猜测方向。

## 八、最终验收标准

1. 学生通过网址登录后，可以独立完成论文中的六阶段实验。
2. 系统具有货物、牵引车、挂车、路线、障碍和绑扎工具的三维教学表达。
3. 高度、圆弧弯道、直交弯道、坡度、桥梁、轴载和绑扎规则能够给出可解释结果。
4. 学生中途刷新或短暂断网后可以恢复实验，不会丢失已提交步骤。
5. 操作提示、关键错误、步骤耗时和结果全部进入操作日志。
6. 系统按照配置权重生成自动评分，教师可以追加评价并导出成绩。
7. 学生无法访问教师功能或其他学生的数据，数据库RLS测试通过。
8. 最新Chrome和Edge中主流程稳定，目标帧率不低于30FPS。
9. 5名首次使用者均能完成实验，阻断性问题为0。
10. 交付物包括源码、正式网址、模型许可清单、数据库迁移、测试报告、学生手册、教师手册和部署说明。

## 九、进度失控时的处理顺序

如果实际进度落后，只允许按以下顺序降级：

1. 降低模型精度和动画复杂度，但保留教学含义。
2. 将自由拖拽改成点选或吸附操作，但保留选择与判断。
3. 将复杂物理过程改成参数计算加预设动画，但明确标注教学简化。
4. 减少场景装饰、音效和非必要过渡动画。
5. 绝不删除六阶段主流程、日志、评分、教师评价、权限和数据恢复。

计划执行完成后，系统应达到“可供一个班级进行教学试用”的标准，而不是仅供答辩观看的演示页面。
